# 后端架构设计

---

## 1. 设计目标

后端负责房间、玩家连接、权威游戏状态、AI 玩家调度和后台预设管理。CLI 是客户端入口，可以创建房间、搜索房间、输入房间号加入，也可以在创建房间或房间大厅中选择 AI 玩家预设加入游戏。

必须支持：
- **网络多人**：不同设备的玩家通过同一后端进行对战
- **CLI 入口**：CLI 能创建房间、搜索公开房间、通过房间号加入、准备、开始、提交行动
- **人机对战**：AI 玩家通过服务端内部 LLM 决策服务参与游戏
- **房间聊天**：人类玩家可以发送房间聊天；AI 聊天回复作为独立 LLM 调用，不与行动决策混用
- **任意混合**：一个房间内任意组合人类玩家和 AI 玩家，人数为 2~4
- **后台管理**：管理员维护 provider、模型预设、AI 玩家预设，并测试连接和单步行动

核心原则：
- `GameEngine` 只处理游戏规则，不依赖 HTTP、WebSocket、CLI、LLM
- 联机模式下，权威 `GameState` 只存在于后端
- CLI 只展示状态和提交意图，不本地执行游戏规则
- AI 玩家不走客户端协议，轮到 AI 时由服务端内部触发
- 房间中保存 AI 玩家预设快照，后台修改不影响进行中的对局

---

## 2. 架构总览

```text
CLI Client
  -> HTTP API：创建房间、搜索房间、加入房间、准备、开始、提交行动、发送聊天
  -> WebSocket/SSE：接收房间事件、游戏状态更新、行动结果、聊天消息

Backend
  -> API Layer
    -> RoomService
    -> GameSessionService
    -> TurnCoordinator
      -> GameEngine
      -> AIPlayerController
        -> LLMDecisionService
        -> LLMChatService
          -> LangChainModelFactory
```

模块职责：

| 模块 | 职责 |
|---|---|
| `API Layer` | HTTP 路由、事件订阅、认证、请求校验 |
| `RoomService` | 房间生命周期、房主权限、玩家加入/离开、AI 玩家加入 |
| `GameSessionService` | 创建和保存一场比赛的权威 `GameState`、`revision`、事件日志 |
| `TurnCoordinator` | 串行处理行动、推进回合、触发 AI 玩家行动 |
| `GameEngine` | 纯游戏规则：装弹、发道具、执行行动、胜负判定 |
| `AIPlayerController` | 轮到 AI 玩家时读取预设快照并发起决策；AI 被点名聊天时发起聊天回复 |
| `LLMDecisionService` | 构造行动决策上下文、调用模型、解析行动、重试和保底 |
| `LLMChatService` | 构造聊天回复上下文、调用模型、解析短文本回复、限流和兜底 |
| `EventPublisher` | 将房间事件和玩家可见状态推送给 CLI |
| `AdminPresetService` | 管理 provider、模型预设、AI 玩家预设和测试连接 |

## 3. 通信边界

### 3.1 HTTP 与事件通道分工

房间管理建议使用 HTTP。事件推送使用 WebSocket 或 SSE。

| 类型 | 用途 |
|---|---|
| HTTP | 创建房间、搜索房间、加入房间、准备、开始、提交行动、发送聊天、添加 AI 玩家、后台管理 |
| WebSocket/SSE | 房间状态变化、游戏事件、行动结果、聊天消息、当前可见状态 |

说明：
- 回合制行动用 HTTP `POST /actions` 足够清晰，便于权限校验和幂等处理
- 聊天可用 HTTP `POST /chat` 或 WebSocket 消息提交；后端统一追加 `chat_message` 事件
- WebSocket/SSE 只负责推送，不作为房间管理的唯一入口
- 纯 HTTP polling 可以作为 MVP，但多人体验和延迟会弱于事件推送

### 3.2 CLI 工作流

CLI 作为游戏入口，需要支持：

```text
启动 CLI
  -> 选择服务器地址
  -> 创建房间 / 搜索房间 / 输入房间号加入
  -> 大厅中准备、添加 AI 玩家、查看玩家列表
  -> 房主开始游戏
  -> 订阅事件并渲染终端界面
  -> 当前玩家提交行动
  -> 输入非命令文本发送聊天
```

CLI 不导入 `GameEngine`，不本地判断隐藏规则，不保存权威状态。它只缓存后端返回的 `PlayerVisibleState` 用于渲染。

## 4. HTTP API 设计

### 4.1 房间 API

| 接口 | 职责 |
|---|---|
| `POST /rooms` | 创建房间 |
| `GET /rooms` | 搜索可加入的公开房间 |
| `GET /rooms/{room_code}` | 查看房间大厅或游戏摘要 |
| `POST /rooms/{room_code}/join` | 输入房间号加入 |
| `POST /rooms/{room_code}/leave` | 离开房间 |
| `POST /rooms/{room_code}/ready` | 切换准备状态 |
| `POST /rooms/{room_code}/start` | 房主开始游戏 |
| `POST /rooms/{room_code}/actions` | 当前行动玩家提交行动 |
| `POST /rooms/{room_code}/chat` | 发送房间聊天 |
| `POST /rooms/{room_code}/ai-players` | 房主添加 AI 玩家 |
| `DELETE /rooms/{room_code}/players/{player_id}` | 房主移除大厅玩家或 AI 玩家 |
| `GET /rooms/{room_code}/events` 或 `WS /rooms/{room_code}/events` | 订阅事件 |

### 4.2 AI 玩家预设 API

普通 CLI 用户只能读取启用的 AI 玩家预设：

| 接口 | 职责 |
|---|---|
| `GET /ai-player-presets` | 获取可加入房间的 AI 玩家预设 |

管理员接口详见 `LLMAPI需求设计.md`，包括 provider、模型预设、AI 玩家预设的增删改查和测试。

### 4.3 行动提交

```json
{
  "player_token": "token",
  "revision": 17,
  "action": {
    "type": "use_item",
    "item": "MAGNIFYING_GLASS"
  }
}
```

规则：
- 只有当前行动玩家能提交行动
- 请求中的 `revision` 必须匹配当前房间状态，旧请求直接拒绝
- 后端必须重新计算合法行动，不能信任 CLI 传来的 `legal_actions`
- 执行行动后由 `TurnCoordinator` 判断是否继续当前玩家、切换玩家、进入下一轮、结束本局游戏或结束整场比赛

### 4.4 聊天提交

```json
{
  "player_token": "token",
  "message": "@DeepSeekV4Pro 你怎么看？"
}
```

规则：
- 房间内玩家都可以发送聊天，聊天不要求当前行动权。
- 聊天追加为 `chat_message` 事件并广播给房间内玩家。
- 聊天事件可以进入房间事件日志和 CLI 事件流，但默认不能进入 AI 行动决策上下文。
- 如果消息显式点名 AI，例如 `@AI名` 或 `@all`，后端可以异步触发 AI 聊天回复。
- AI 聊天回复只追加聊天事件，不推进游戏规则，不提交行动，不修改权威 `GameState`。

## 5. 数据模型

### 5.1 Room

```python
Room:
  id: str
  room_code: str              # 给 CLI 输入加入用的短号码
  name: str
  visibility: PUBLIC | PRIVATE
  status: LOBBY | IN_GAME | FINISHED | CLOSED
  owner_player_id: str
  max_players: int            # 2~4
  config: MatchConfig
  players: list[RoomPlayer]
  game_session_id: str | None
  created_at: datetime
  updated_at: datetime
```

### 5.2 RoomPlayer

```python
RoomPlayer:
  id: str                     # 后端玩家会话 id，不等于座位号
  seat_index: int | None      # 游戏开始后固定为 Player.id
  name: str
  type: HUMAN | AI
  status: CONNECTED | DISCONNECTED | READY | LEFT
  token_hash: str | None      # 人类玩家鉴权 token 的服务端哈希
  ai_preset_snapshot: AIPlayerPresetSnapshot | None
  joined_at: datetime
```

规则：
- 游戏开始前，`seat_index` 可以调整
- 游戏开始后，`seat_index` 固定，对应游戏引擎中的 `Player.id`
- AI 玩家没有客户端连接，不需要 WebSocket token

### 5.3 GameSession

```python
GameSession:
  id: str
  room_id: str
  state: GameState            # 权威游戏状态
  revision: int               # 每次状态变更递增
  event_log: list[GameEvent]
  action_lock: Lock
```

同一房间内的行动必须串行执行。`action_lock` 或等价机制用于避免两个请求同时修改同一个 `GameState`。

### 5.4 GameEvent

```python
GameEvent:
  event_id: int
  room_id: str
  game_id: str
  revision: int
  match_index: int | None
  reload_round: int | None
  actor_player_id: int | None
  event_type: str
  payload: dict
  visible_to: list[int] | ALL
  created_at: datetime
```

事件用于 CLI 推送，也作为后端构造 AI 上下文的原始来源。构造 AI 上下文时必须按调用类型过滤事件：行动决策上下文默认排除 `chat_message`，聊天回复上下文只读取聊天消息和必要的公开局面摘要。只对某个玩家可见的道具结果必须设置 `visible_to`。

### 5.5 PlayerVisibleState

```python
PlayerVisibleState:
  room_id: str
  game_id: str
  player_id: str
  revision: int
  room_status: str
  visible_players: list
  visible_items: list
  public_shell_counts: dict
  current_player_id: int | None
  legal_actions: list[Action]
  visible_events: list[GameEvent]
```

人类 CLI 和 AI 玩家都使用同一套 `PlayerVisibleState`。隐藏弹序、其他玩家不可见的信息、服务端配置、API Key 不能进入可见状态。

`PlayerVisibleState.visible_events` 面向客户端展示，可以包含聊天事件。提供给 AI 行动决策的上下文不能直接复用完整 `visible_events`，必须过滤出公开游戏事件和当前可见局面，避免聊天消息影响决策强度或注入策略。

## 6. AI 玩家预设快照

后台管理维护两类预设：
- `ModelPreset`：provider、真实模型名、temperature、max_tokens、reasoning_effort、timeout 等
- `AIPlayerPreset`：AI 名称、角色提示词、策略提示词、聊天配置、引用的模型预设、行动上限和保底策略

房间中保存的是快照：

```python
AIPlayerPresetSnapshot:
  preset_id: str
  preset_version: int
  display_name: str
  model_preset_snapshot: ModelPresetSnapshot
  chat_model_preset_snapshot: ModelPresetSnapshot | None
  persona_prompt: str
  strategy_prompt: str
  chat_enabled: bool
  chat_prompt: str
  chat_trigger_mode: str
  chat_max_chars: int
  chat_cooldown_seconds: int
  max_item_actions_per_turn: int
  max_parse_failures_per_turn: int
  max_illegal_actions_per_turn: int
  fallback_policy: str

ModelPresetSnapshot:
  preset_id: str
  preset_version: int
  provider_id: str
  protocol: str
  model_name: str              # 真实模型名
  temperature: float | None
  max_tokens: int | None
  reasoning_effort: str | None # provider-aware 字段
  timeout_seconds: int
  max_retries: int
  extra: dict
```

快照规则：
- CLI 创建房间或添加 AI 玩家时只传 `ai_player_preset_id`
- 后端读取 `AIPlayerPreset` 和其引用的决策 `ModelPreset`，必要时读取聊天 `ModelPreset`，生成快照写入 `RoomPlayer`
- 后台编辑预设只影响之后新建的 AI 玩家
- 进行中的房间不跟随后台预设变化
- 默认聊天复用决策模型。只有配置了 `chat_model_preset_id` 时，才使用独立聊天模型。

## 7. 回合调度

### 7.1 人类玩家行动

```text
CLI POST /rooms/{room_code}/actions
  -> API 校验 token、room、revision
  -> TurnCoordinator 校验当前行动权
  -> 重新生成合法行动并校验 action
  -> GameEngine.execute_action()
  -> 追加 GameEvent
  -> revision + 1
  -> EventPublisher 推送 action_result 和 PlayerVisibleState
  -> 如果下一位是 AI，触发 AIPlayerController
```

### 7.2 AI 玩家行动

```text
TurnCoordinator 发现当前玩家是 AI
  -> AIPlayerController 读取 RoomPlayer.ai_preset_snapshot
  -> LLMDecisionService 构造当前可见局面 + 公开游戏事件上下文
  -> 调用模型并解析单步行动
  -> 通过同一套 ActionValidator / GameEngine 执行
  -> 追加事件并推送给客户端
  -> 如果行动权仍是同一个 AI，继续请求下一步
```

AI 不直接调用 HTTP API，也不走 WebSocket。行动决策模型只能返回结构化行动，不能绕过后端校验。

行动决策上下文只包含：
- 基础信息：玩家、座位、AI 自身配置、每局游戏初始化信息。
- 公开游戏事件：装弹、发道具、道具使用、射击结果、回合变化、出局和胜负。
- 当前可见状态：血量、道具、当前行动权、公开弹匣数量、合法行动。

行动决策上下文默认不包含：
- `chat_message`。
- 私有道具结果中对该 AI 不可见的内容。
- 服务端额外推导的胜率、概率建议、玩家画像结论。
- API Key、provider 完整配置、系统路径。

聊天消息如果未来要影响行动决策，必须作为显式 AI 预设开关和策略能力，而不是默认输入。

### 7.3 AI 聊天回复

```text
玩家发送聊天
  -> TurnCoordinator 追加 chat_message 事件并推送
  -> 如果消息 @AI名 或 @all，筛选启用聊天的 AI 玩家
  -> AIPlayerController 读取 RoomPlayer.ai_preset_snapshot
  -> LLMChatService 构造聊天上下文
  -> 调用模型并解析短文本 reply
  -> 追加 AI chat_message 事件并推送
```

AI 聊天上下文只包含：
- `persona_prompt` 和 `chat_prompt`。
- 最近有限条聊天消息。
- 被点名的触发消息。
- 必要的公开局面摘要和少量最近公开游戏事件。

AI 聊天上下文不包含：
- 合法行动列表。
- 隐藏弹序、不可见道具结果和内部调试状态。
- 完整 provider 配置和 API Key。

AI 聊天回复必须满足：
- 不提交行动，不改变 `GameState`，不推进回合。
- 有最大长度、冷却时间和并发保护。
- 解析失败、空回复或模型错误时不影响游戏流程。
- 同一个 AI 正在进行行动决策时，聊天请求可以延迟或丢弃，不能打断行动循环。

### 7.4 过期结果处理

LLM 调用可能慢于房间状态变化，所以 AI 决策结果落库前必须检查：
- 房间仍存在且未关闭
- `GameSession.revision` 仍是决策开始时的 revision
- 当前行动玩家仍是同一个 AI
- 本局游戏未结束

任一条件不满足，丢弃该 LLM 结果。

AI 聊天回复落库前也必须检查房间仍存在、AI 玩家仍在房间中、触发消息仍可见；不需要检查当前行动权。

## 8. 事件推送

服务端每次状态变化后推送两类内容：

| 事件 | 用途 |
|---|---|
| `room_updated` | 大厅玩家、准备状态、AI 玩家列表变化 |
| `game_started` | 比赛开始 |
| `round_started` | 新一轮装弹和公开 LIVE/BLANK 数量 |
| `action_result` | 行动执行结果 |
| `turn_changed` | 当前行动权变化 |
| `chat_message` | 房间聊天消息，包含人类聊天和 AI 聊天回复 |
| `match_ended` | 单局游戏结束 |
| `game_ended` | 整场比赛结束 |
| `visible_state` | 针对当前订阅玩家的最新可见状态 |
| `error` | 请求或连接错误 |

推荐每次状态变化后推送 `visible_state`，CLI 用全量可见状态覆盖本地缓存，避免复杂增量合并。

## 9. CLI 客户端

CLI 是薄客户端：
- 不导入 `GameEngine`
- 不本地执行规则
- 不保存隐藏状态
- 通过 HTTP 发起房间和行动请求
- 通过 WebSocket/SSE 接收 `visible_state` 并渲染

CLI 菜单建议：

```text
1. 创建房间
2. 搜索公开房间
3. 输入房间号加入
4. 查看 AI 玩家预设
5. 退出
```

房间大厅建议：

```text
1. 准备 / 取消准备
2. 添加 AI 玩家
3. 移除 AI 玩家或空座位
4. 开始游戏（房主）
5. 离开房间
```

## 11. 阶段化实现计划

### 第一阶段：本地引擎稳定

- 完成 `domain/models.py` 和 `domain/engine.py`
- 用单元测试覆盖装弹、道具、行动、回合切换、一局游戏结束
- 不接入 HTTP、CLI 联机或 LLM

### 第二阶段：房间和 CLI 联机

- 实现 HTTP 房间管理
- 实现事件推送
- CLI 支持创建房间、搜索房间、输入房间号加入
- CLI 支持提交人类玩家行动

### 第三阶段：AI 玩家预设接入

- 实现后台 AI 玩家预设读取
- CLI 可选择 AI 玩家预设加入房间
- `TurnCoordinator` 轮到 AI 时触发 `AIPlayerController`
- 使用 fake LLM 验证 AI 行动循环

### 第四阶段：真实 LLM Provider

- 按 `LLMAPI需求设计.md` 实现 provider 和模型预设管理
- 实现 `model-preset test` 和 `ai-preset test-action`
- 接入真实 OpenAI/Anthropic/Gemini/DeepSeek 或第三方 provider
